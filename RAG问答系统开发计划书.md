# RAG 问答系统开发计划书

## 一、项目概述

基于架构图，建设一套分层检索 + 意图识别 + RAG 生成的智能问答系统。用户提问后依次经过缓存命中、关键词检索、数据库检索、意图分类，对专业问题走向量检索与重排序，最终由大模型生成回答。同时通过 Playwright 浏览器自动化实现多平台客服消息监控与 AI 回复，支持飞鸽、视频号、快手、小红书等平台。

**开发周期：** 2 周（10 个工作日）  
**开发人员：** 2 人（并行开发）

---

## 二、技术栈（大致）

| 层级 | 技术选型 |
|------|----------|
| **消息接入** | Playwright 浏览器自动化（模拟人工操作客服后台） |
| | 多平台适配器（飞鸽 / 视频号 / 快手 / 小红书），Cookie 持久化复用登录态 |
| **接入层** | FastAPI / Flask，RESTful API |
| **缓存** | Redis（问答缓存、热点命中） |
| **关键词检索** | BM25（如 rank_bm25 / Elasticsearch） |
| **结构化数据** | MySQL（业务问答库、同义词/标准问） |
| **意图识别** | BERT 微调分类模型（LLaMA-Factory / HuggingFace） |
| **大语言模型** | Qwen3-4B-Instruct（本地部署 / vLLM / Ollama） |
| **向量化** | BGE-M3（稠密 + 稀疏向量） |
| **向量数据库** | Milvus（混合检索、父子文档） |
| **重排序** | BGE-Reranker-Large |
| **策略与改写** | LLM API（策略选择、假设性问题、子问题拆解、回溯问题） |
| **编排与部署** | Python 3.11+，Docker（可选），环境变量配置 |

## 三、人员分工（并行）

| 开发人员 | 负责模块 | 说明 |
|----------|----------|------|
| **开发 A** | 接入层 + 缓存 + BM25 + MySQL + 意图识别 + 通用问答链路 | 环节 1～5：请求入口、Redis、BM25、MySQL、BERT 分类及非专业问题直出 LLM |

| 工作日 | 开发 A | 开发 B |

## 六、风险与依赖（简要）

- **模型与数据**：BERT 意图模型、BGE-M3、Reranker、Qwen 需提前准备好权重与推理环境。  
- **Milvus / Redis / MySQL**：D1 前完成基础环境可用。  
- **页面 DOM 结构变更**：各平台客服后台 UI 更新可能导致选择器失效，需预留选择器热更新或容错机制。  
- **登录态失效**：Cookie / Session 有效期不定，需检测失效后自动暂停并告警，等待手动重新登录。  
- **平台风控**：模拟人工操作间隔过短可能触发反爬，需加入随机延迟、打字模拟等行为。  
- **多会话冲突**：多个用户同时发消息时需排队或并发处理，避免回复错乱。  
- **并行协作**：D1 约定 API 与意图枚举；D5、D7 固定联调日，避免后期集成阻塞。
